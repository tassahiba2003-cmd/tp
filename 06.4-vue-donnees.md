# SupEvents – Modèle de données, architecture et conformité
## §6.4 — Vue des données

### Modèle entité-relation
Ce bloc de conception technique pour SupEvents définit la structure sémantique des données persistées par le système. Contrairement à un simple schéma physique, cette section explique les décisions architecturales qui garantissent l'interopérabilité des services[cite: 1]. Nous documentons ici le modèle logique qui supporte les flux transactionnels critiques comme les inscriptions et les paiements. L'objectif est de rendre visibles les invariants métier pour faciliter les développements futurs et la maintenance du système.

## 1. Vue d’ensemble

L’entité **centrale** du système est le **`TICKET`**, qui matérialise l’inscription et lie l’utilisateur à l’événement.  
• Chaque ticket possède ses propres attributs de **prix** et de **statut** (`pending`, `confirmed`, `cancelled`).  
• La jointure simple devient ainsi une **entité métier** capable de gérer un cycle de vie complexe.

L’authentification est **déléguée** au SSO OIDC de l’école :  
• L’entité **`USER`** ne stocke **aucun mot de passe** local → surface d’attaque réduite.

Les données de paiement sont isolées :  
• `payment_intent_id` (Stripe) assure la conformité **PCI-DSS** ; aucune donnée bancaire sensible n’est conservée.

## 2. Dictionnaire de données

### 2.1 `User`

| Champ            | Type          | Contraintes       | Description                                         | Sensibilité RGPD |
|------------------|--------------|-------------------|-----------------------------------------------------|------------------|
| `id`             | UUID         | PK, NOT NULL      | Identifiant technique unique                        | Non              |
| `sso_subject_id` | VARCHAR(255) | UK, NOT NULL      | Identifiant pivot fourni par le SSO OIDC            | Oui              |
| `email`          | VARCHAR(255) | UK, NOT NULL      | Adresse e-mail institutionnelle                     | Oui              |
| `role`           | VARCHAR(50)  | NOT NULL          | Rôle utilisateur (`student`, `organizer`, `admin`)  | Non              |
| `created_at`     | TIMESTAMP    | NOT NULL          | Date de création du compte                          | Non              |

### 2.2 `Event`

| Champ         | Type         | Contraintes                     | Description                                    | Sensibilité RGPD |
|---------------|--------------|---------------------------------|------------------------------------------------|------------------|
| `id`          | UUID         | PK, NOT NULL                    | Identifiant unique de l’événement              | Non              |
| `organizer_id`| UUID         | FK → `Organizer.id`            | Référence vers l’organisateur                  | Non              |
| `title`       | VARCHAR(200) | NOT NULL                        | Titre de l’événement                           | Non              |
| `start_date`  | TIMESTAMP    | NOT NULL                        | Date et heure de début                         | Non              |
| `status`      | VARCHAR(20)  | CHECK (`draft`,`pub`,`cancel`)  | État de publication                             | Non              |

### 2.3 `Ticket`

| Champ      | Type          | Contraintes          | Description                                   | Sensibilité RGPD |
|------------|---------------|----------------------|-----------------------------------------------|------------------|
| `id`       | UUID          | PK, NOT NULL         | Identifiant unique du billet                  | Non              |
| `user_id`  | UUID          | FK → `User.id`       | Détenteur du billet                           | Non              |
| `event_id` | UUID          | FK → `Event.id`      | Événement associé                             | Non              |
| `status`   | VARCHAR(20)   | NOT NULL             | `pending`, `confirmed`, `cancelled`           | Non              |
| `price`    | NUMERIC(10,2) | NOT NULL, `>= 0`     | Prix d’achat lors de la transaction           | Non              |

### 2.4 `Payment`

| Champ               | Type          | Contraintes        | Description                           | Sensibilité RGPD |
|---------------------|---------------|--------------------|---------------------------------------|------------------|
| `id`                | UUID          | PK, NOT NULL       | Identifiant technique de la transaction| Non              |
| `ticket_id`         | UUID          | FK → `Ticket.id`   | Billet rattaché                       | Non              |
| `payment_intent_id` | VARCHAR(255)  | UK, NOT NULL       | Référence Stripe (token)              | Non              |
| `amount`            | NUMERIC(10,2) | NOT NULL           | Montant total payé                    | Non              |

### 2.5 `Organizer`

| Champ          | Type          | Contraintes   | Description                            | Sensibilité RGPD |
|----------------|---------------|---------------|----------------------------------------|------------------|
| `id`           | UUID          | PK, NOT NULL  | Identifiant technique                  | Non              |
| `company_name` | VARCHAR(100)  | NOT NULL      | Nom de l’association / entité          | Non              |
| `status`       | VARCHAR(20)   | NOT NULL      | Statut de validation par l’admin       | Non              |

## 3. Stratégie de stockage

| Composant  | Rôle principal                                                                              |
|------------|---------------------------------------------------------------------------------------------|
| **PostgreSQL** | Base transactionnelle (ACID) pour `User`, `Event`, `Ticket`, `Payment`. Assure la cohérence lors du **décrément de stock** et de la **capture de paiement**. |
| **Redis**      | Sessions utilisateurs + **idempotence** des opérations critiques. Latence sub-milliseconde. |
| **S3 / MinIO** | Stockage horizontalement scalable des médias (affiches, billets PDF).                     |
| **RabbitMQ**   | Bus d’événements asynchrone : envoie vers SendGrid & autres services, évitant les blocages. |

---

## 4. Conformité RGPD & sécurité

* Données **personnelles identifiantes** : `email`, `sso_subject_id` (marquées « Oui » dans le dictionnaire).  
* **Aucune** donnée bancaire sensible (n° de carte, CVC) n’est stockée ; la **PCI-DSS** est déléguée à **Stripe**.  
* Données personnelles **chiffrées au repos** ; **pseudonymisation** appliquée sur demande (droit à l’oubli).
